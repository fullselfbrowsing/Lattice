<verification_findings>
Pre-plan code verification completed by planner. Summary of findings:

**P1 CONFIRMED real bug.**
`createOpenAICompatibleRequestBody` (adapters.ts line 296-357) spreads
`{ stream: true }` when `input.stream === true` (line 355) but never adds
`stream_options: { include_usage: true }`. The function is shared by
`createOpenAICompatibleProvider` AND `createOpenAIProvider` (which delegates
to `createOpenAICompatibleProvider` via `innerCompat` at line 1009). So OpenAI,
xAI, OpenRouter, LM Studio, and LiteLLM-gateway are all affected.
The existing test at adapters.test.ts:385 checks `body.stream === true` but does
not assert `body.stream_options`.

**P2-1 CONFIRMED real bug.**
`ReceiptProjection` (diff.ts line 39-71) omits: `contractVerdict`, `contractHash`,
`issuedAt`, `modelClass`, `redactionPolicyId`, `redactions`, `noRouteReasons`,
`tripwireEvidence`, and step-marker fields (`stepName`, `stepIndex`,
`parentStepName`, `previousStepName`, `sessionId`, `timestamp`).
`compareProjection` only walks the projected subset, so two receipts differing
only in `contractVerdict` ("success" vs "tripwire-violated") compare equal.
Confirmed: receipt-diff.test.ts has no test that catches a `contractVerdict`-only
difference.

**P2-2 NOT A BUG — no code change.**
`noPublicUrl` in packaging.ts (line 132) blocks transport mode `"url"` which
is the HTTP/HTTPS URL path. The Gemini `file-id` transport reads `geminiFileUri`
from artifact metadata — these are Gemini-internal storage handles (e.g.
`files/audio-123`), not public URLs. The `"url"` transport branch in gemini.ts
(line 184) which passes HTTP URLs is already blocked by `chooseTransport` when
`noPublicUrl=true`, so no HTTP URI reaches `createGeminiUserParts`. The `gs://`
claim in the review conflates Gemini file API URIs with publicly reachable URLs —
they are not. No fix warranted; document this analysis in a comment.

**P2-3 CONFIRMED real gap.**
`lineageArtifacts` passed to `maybeIssueReceipt` at create-ai.ts line 674 is
`[...built.artifacts, ...artifactRefs]`. `attemptPackaging.packagedArtifacts`
(derived `ArtifactRef` objects with `lineage.parents` pointing to input artifacts
and a `provider-packaging` transform) are NOT included. Since
`computeArtifactLineageMerkleRoot` only folds in artifacts where
`artifact.lineage !== undefined`, and packaged artifacts carry lineage, their
omission means the receipt's `lineageMerkleRoot` does not commit to the
provider-specific packaging transform applied to the inputs — a lineage gap.
</verification_findings>
