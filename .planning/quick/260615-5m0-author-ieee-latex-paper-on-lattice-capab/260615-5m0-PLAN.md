   systems; why provider-wrapper SDKs do not close it; the four guarantees Lattice

   non-repudiation via signature, redaction before signing (no secret ever enters

### Receipts (packages/lattice/src/receipts/)
- RFC 8785 JCS canonicalization via `canonicalize@3.0.0` (`canonical.ts`,
  `canonicalizeReceiptBody`).
- Ed25519 via Node 24 WebCrypto `crypto.subtle` with the literal algorithm name
  `"Ed25519"` (`sign.ts`). Signatures are 64 bytes (RFC 8032). Keys are JWK OKP.
  `@noble/ed25519@3.1.0` is used only in tests as a byte-for-byte parity oracle to
  catch silent V8 regressions; it is not in the production signing path.
- DSSE v1.0 envelope (`envelope.ts`): `payloadType =
  "application/vnd.lattice.receipt+json"`, base64 `payload`, `signatures: [{ keyid, sig }]`.
  PAE = `UTF-8("DSSEv1 " + len(payloadType) + " " + payloadType + " " + len(payload) + " " + payload)`,
  lengths in decimal ASCII, payload base64. The signature is computed over the PAE
  bytes, not the raw JSON.
- `kid` and KeySet (`types.ts`, `keyset.ts`): `KeyState = "active" | "retired" | "revoked"`;
  `KeySet.lookup(kid)`; verify checks key state before crypto; `revoked` yields
  `key-revoked`; a defense-in-depth check rejects `body.kid !== entry.kid` as
  `signature-invalid`.
- Redact-then-sign ordering is structurally enforced (`receipt.ts`,`redact.ts`):
  `canonicalizeReceiptBody` is only ever called on the output of `redactReceiptBody`,
  so any other order is unrepresentable. The signed body carries a `redactions[]`
  manifest of `{ path, reason }` and a signed `redactionPolicyId` (default
  `"lattice.default.v1"`).
- `verifyReceipt` returns `VerifyResult = VerifyOk { ok: true, body, keyState } |
  VerifyFail { ok: false, error }`. There are exactly SEVEN `VerifyErrorKind` values:
  `key-not-found`, `key-revoked`, `canonicalization-mismatch`, `signature-invalid`,
  `envelope-malformed`, `version-mismatch`, `schema-version-too-low`.
- Schema versions: `lattice-receipt/v1`, `/v1.1`, `/v1.2`. v1.2 adds an optional
  signed `modelClass` (`TrainingClass`). Downgrade defense (CRYPTO-01): a v1 or
  version-undefined body is rejected with `schema-version-too-low` BEFORE any
  cryptographic work, so an attacker with a valid key cannot mint a v1-shaped body
  to bypass newer schema commitments.
- `costUsd` is serialized as an I-JSON string or null inside the signed body
  (`ReceiptUsageCanonical { promptTokens, completionTokens, costUsd: string | null }`),
  via `stringifyCostUsd`; receipts never carry raw floats (RFC 8785 / I-JSON).
- Every receipt records a `ContractVerdict`: `success | tripwire-violated |
  no-contract-match | execution-failed | validation-failed`.

### Replay (packages/lattice/src/replay/)
- `ReplayEnvelope.receipt?` and `ReplayEnvelope.contract?` are additive optional
  fields. `materializeReplayEnvelope` verifies the receipt BEFORE loading any
  artifact; a test asserts the artifact loader is called zero times when verify
  fails. `MaterializationError` has THREE kinds: `verify-failed`,
  `artifact-load-failed`, `envelope-malformed`. A property test asserts
  `createReceipt -> materializeReplayEnvelope -> replayOffline` preserves the
  `outputHash`.

### Router, tripwires, contracts (packages/lattice/src/routing|contract/)
- `routeDeterministically` is a pure function: no randomness, no model calls. Hard
  filters reject on availability, forced provider/model mismatch, input/output
  modality unsupported, structured-output unsupported, context-window exceeded.
  Policy filters reject on allow/deny list, privacy, no-logging, no-upload,
  latency-class, budget. Scoring combines cost, latency, context headroom, and
  catalog index, with a stable tie-break (acceptance, score, provider name, model
  name). `RouteDecision` exposes `selected?`, `candidates[]`, `rejected[]`,
  `fallbackChain[]`, and a deduplicated `noRouteReasons[]`. `ExecutionPlan` is stable
  JSON with a nine-stage pipeline: analysis, transforms, context-packing,
  provider-packaging, tool-execution, execution, validation, tripwire, persistence.
- Tripwire invariants: the fluent `inv` builder exposes `mustCite`, `fieldFromTable`,
  `noPII`, and `matches` (the last over a Standard Schema). `evaluateTripwires` is a
  pure kernel (no I/O, no `Date.now`, no randomness) that aborts on the first
  violation. `defaultPiiDetectors` has FOUR detectors: email, US SSN, credit card
  (Luhn validated), US phone. A `no-pii` violation redacts evidence to
  `{ detector, substring }` so the secret never lands in the evidence. A
  `TripwireViolationError` carries `terminal: true`; `isTerminal()` returns true for
  `tripwire-violated` and `no-contract-match`, so the fallback chain does NOT retry a
  terminal failure.
- Contracts: `contract({ budget, invariants, qualityFloor, requiredModalities,
  requiredPrivacy })` is optional on `ai.run`. A pure preflight evaluator
  (`evaluateContractAgainstRoute`) runs inside the router and can reject with
  `contract-budget-exceeded`, `contract-modality-missing`, `contract-privacy-mismatch`
  (and reserves `contract-quality-floor`). `Usage { promptTokens, completionTokens,
  costUsd: number | null }` is normalized on every success AND failure;
  `costUsd: null` means unmeasured, distinct from zero. Capability entries carry
  per-1k-token pricing.

### Providers, registry, CLI
- Seven logical provider adapters, proven equal by the INV-03 parity smoke test
  (`providers/parity.test.ts`): `openai`, `openai-compatible`, `anthropic`, `gemini`,
  `xai`, `openrouter`, `lm-studio`. INV-03 means "every improvement works equally
  across all seven universal-provider targets." The parity test asserts uniform
  adapter shape, string `rawOutputs`, the normalized usage shape, error-on-non-OK
  fetch, `AbortSignal` propagation, raw response pass-through, and that the set of
  ids has size seven. (A separate `createAISdkProvider` integration exists but is not
  one of the seven INV-03 targets.)
- Capability registry: 332 profiles total = 328 generated from an OpenRouter feed
  snapshot (`registry.generated.ts`) plus 4 hand-maintained static supplements
  (`registry.static.ts`: anthropic `claude-opus-4`, gemini `gemini-2.5-pro`,
  lm-studio `local-template`, xai `grok-4`). A weekly CI drift gate guards the
  generated file. Adapter quirk flags and capability negotiation (live `/models`
  with registry fallback, TTL cache, inflight coalescing) refine the profiles.
- `lattice` CLI (`packages/lattice-cli`, citty, lazy-loaded subcommands, exit codes
  0/1/2): `repro` (load -> verify -> materialize -> replayOffline -> diff outputHash;
  0 match, 1 drift, 2 load failure), `verify` (signature + structural, prints
  `OK kid=... verdict=...`), `eval` (walks receipts, replays, gates layered
  determinism and baseline-relative cost/quality regressions, `--init-baseline`,
  judge cache). IMPORTANT: `eval --agent` and a `receipt diff` subcommand are NOT
  implemented yet; they are v1.4 future work. Do not claim they exist.

### Multi-agent surface and examples
- `ai.runAgent(intent)` drives a uniform prompt-reencoded tool-use loop across the
  seven providers (v1.2). `defineAgent` and `runAgentCrew` add an opt-in crew surface
  (v1.3) with crew budgets (`maxTotalIterations`, `maxIterationsPerAgent`,
  `maxConcurrentChildren` default 1, `maxDepth` default 1), prompt cache-prefix
  sharing, rate-limit groups (`createRateLimitGroup`, default 50 rpm / 30000 tpm),
  and `parentReceiptCid`-chained receipts. `evalAgentRun` gates iterations-to-goal
  and cost against a baseline.
- Three runnable examples: `work-inbox` (v1.1 contract, signed receipt, tripwire,
  no-contract-match), `agent-loop` (v1.2 single agent, per-iteration checkpoint
  receipts), `agent-crew` (v1.3 crew, `parentReceiptCid` chaining, eval gate).
