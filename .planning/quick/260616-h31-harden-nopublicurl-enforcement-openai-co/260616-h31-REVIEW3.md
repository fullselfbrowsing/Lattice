The patched OpenAI-compatible adapter suppresses the raw URL, so built-ins look safe, but the packaging plan can say “base64/provider-upload” even when the only source is a public URL. - **P3** [adapters.ts](/Users/lakshman/conductor/workspaces/lattice/dubai/packages/lattice/src/providers/adapters.ts:331): direct public adapter calls that omit `providerPackaging` now drop URL artifacts because `resolvedTransport === undefined` is treated as non-URL. This is privacy-safe, but it is a compatibility regression if direct `createOpenAICompatibleProvider().execute()` artifact calls are supported.

**Assessment**

The patched `adapters.ts` gating is correct for runtime-packaged OpenAI-compatible requests: `url` only emits for `transport === "url"`, public URL `value` fields are dropped for non-URL transports, data URLs are not treated as public HTTP URLs, and lookup misses fail closed.

The new tests would fail if the OpenAI-compatible URL/value gating were removed, and Anthropic/Gemini parity locks cover the base64 media branches. They do not cover gateway metadata URLs, malformed base64 metadata, direct calls without packaging, or the stated positive case that ordinary non-URL string `value` content remains serialized.

I would not call `noPublicUrl` enforcement complete yet because of the metadata and base64-string paths above.
