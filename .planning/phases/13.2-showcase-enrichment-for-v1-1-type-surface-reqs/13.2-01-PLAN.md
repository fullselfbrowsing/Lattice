**File 1 (new): `examples/work-inbox/judges.mjs`**
Export `stubScoringJudge(scores)` — a factory that returns `{ judge, get callCount() }` where `judge` conforms to the `Judge` interface (`score(input): Promise<number>`). The factory closes over the `scores` array and a 0-based index. Each `score()` call:
  1. Increments `callCount`.
  2. Returns `scores[index % scores.length]` (cyclic).
  3. Increments `index`.
The judge MUST be deterministic and stateless w.r.t. `input` (the scoring schedule is determined only by call order, not by `input.fixtureId` or `input.output`). This is intentional so the cache-key hash is stable across runs (CONTEXT.md "the judge identity is in the judgePrompt string used as a cache key").

Also export a constant `STUB_JUDGE_PROMPT = "lattice-stub-judge-v1"` for reuse across showcase + Plan 02 tests so the cache key is identical on both sides.

File header (zero-emoji, per global CLAUDE.md): one-line summary plus a 3-line NOTE explaining: the closure with state lives in the test/showcase process; the receipt/sidecar never serialize this closure; the cyclic scoring schedule lets Plan 02 verify N=3 median (`[0.6, 0.8, 0.7]` → 0.7) and cache-hit semantics (second eval call must NOT advance the counter past the first call's 3 calls).
