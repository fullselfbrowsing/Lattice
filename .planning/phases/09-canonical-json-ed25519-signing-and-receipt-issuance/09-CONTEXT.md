# Phase 9: Canonical JSON, Ed25519 Signing, and Receipt Issuance - Context

**Gathered:** 2026-05-11
**Status:** Ready for planning

<domain>
## Phase Boundary

Every `ai.run` (success or failure) issues a `CapabilityReceipt` that is RFC 8785-canonicalized (JCS), signed over the redacted form with Node 24 WebCrypto Ed25519 in a DSSE-shaped envelope, carries `kid` plus `model.observed` fingerprint, and can be verified by a pure `verifyReceipt` against a `KeySet`. Phase 9 is the highest-risk phase — decisions made here cannot be retrofitted once receipts are in the wild.

Out of scope for Phase 9: ReplayEnvelope embedding (Phase 10), CLI (Phase 11), eval gate (Phase 12). Receipt-aware verification is implemented; receipts inside replay envelopes is Phase 10.
</domain>

<decisions>
## Implementation Decisions

### Cryptographic Choices (UNRETROFITTABLE)
- **Ed25519 via Node 24 WebCrypto** — `crypto.subtle` with algorithm `"Ed25519"`. Zero new runtime dependency. Isomorphic with browsers, Workers, Deno Deploy for a future verify-only subpath export.
- **DSSE-shaped envelope** — `{ payloadType: "application/vnd.lattice.receipt+json", payload: base64(canonical_json), signatures: [{ keyid, sig }] }`. PAE pre-auth encoding: `"DSSEv1 " + len(payloadType) + " " + payloadType + " " + len(payload) + " " + payload`. ~80 LOC against WebCrypto + canonicalize; NO `@sigstore/*` deps.
- **`canonicalize@3.0.0`** for RFC 8785 JCS. Pin exact version. Add as a runtime dep to `packages/lattice/package.json`. `@noble/ed25519@3.1.0` as devDep ONLY for parity oracle tests.

### Redact-Then-Sign Ordering (UNRETROFITTABLE)
- The signed digest commits to `canonicalize(redact(receipt_body))`. Redaction ALWAYS runs before canonicalization, ALWAYS before signing.
- The signed payload includes a `redactions[]` manifest declaring `[{ path: string, redactionPolicyId: string }]` for every elided field.
- `redactionPolicyId` is a free-form string in v1.1 (registry enforcement deferred to v1.2). Default policy id is `"lattice.default.v1"`.
- The receipt redactor lives in `packages/lattice/src/receipts/redact.ts`. The existing v1.0 `replay/replay.ts` redactor is referenced for behavior parity but Phase 9 ships its OWN redactor specialized for receipts (no path coupling — preserves receipt evolution independence).

### Key Management (UNRETROFITTABLE)
- `kid` is REQUIRED in every receipt signature. The signed payload also embeds `kid` so it cannot be silently swapped.
- `KeySet` interface with three states: `active | retired | revoked`. Verifier looks up `kid` and refuses signatures from `revoked` keys; warns on `retired`.
- `ReceiptSigner` interface: `{ kid: string, sign(bytes: Uint8Array): Promise<Uint8Array>, publicKeyJwk: JsonWebKey }`. The runtime accepts a signer reference, NEVER raw private keys.
- `LatticeConfig.signer?: ReceiptSigner` — when absent, receipts are NOT emitted. When present, receipts are emitted on BOTH success and failure runs.
- Built-in `createInMemorySigner(privateKeyJwk)` factory exported for tests and dev. Production users implement their own signer (KMS adapter, OS keyring, etc.) — KMS adapter shapes deferred to v1.2.

### Receipt Schema (UNRETROFITTABLE)
The `CapabilityReceipt` body (the thing that gets canonicalized + signed) contains:
- `version: "lattice-receipt/v1"` — schema version tag
- `receiptId: string` — random UUID v4 generated by the runtime (not signed-content-derived; pure id)
- `runId: string` — corresponds to the `RunIntent` id (or generated if absent)
- `issuedAt: string` — ISO 8601 UTC timestamp
- `kid: string` — duplicated inside the signed body (defense in depth)
- `model: { requested: string, observed: string | null }` — `requested` is what the developer asked for; `observed` is the provider's fingerprint (when available, else null)
- `route: { providerId: string, capabilityId: string, attemptNumber: number }`
- `usage: Usage` — directly from the Phase 7 normalized usage shape
- `contractVerdict: "success" | "tripwire-violated" | "no-contract-match" | "execution-failed" | "validation-failed"`
- `contractHash: string | null` — SHA-256 hex of `canonicalize(contract)` when a contract was attached; null otherwise
- `inputHashes: string[]` — SHA-256 hex of each input artifact's canonical form
- `outputHash: string | null` — SHA-256 hex of `canonicalize(redact(output))` on success; null on failure
- `redactionPolicyId: string` — see Redact-Then-Sign Ordering
- `redactions: ReadonlyArray<{ path: string, reason: string }>` — paths elided during redaction
- `noRouteReasons?: RouteRejectReason[]` — only present when `contractVerdict === "no-contract-match"`
- `tripwireEvidence?: TripwireEvidence` — only present when `contractVerdict === "tripwire-violated"`

I-JSON only — no raw floats anywhere. `costUsd` in `usage` is the only float-shaped field; serialize as string when writing to receipts (`"0.000125"`) to keep the canonical form stable across V8 versions. The conversion lives inside the receipt builder, not on `RunResult.usage` (Phase 7 keeps the number representation).

### Verification
- Pure function: `verifyReceipt(envelope: ReceiptEnvelope, keySet: KeySet): VerifyResult`
- `VerifyResult` = `{ ok: true, body: CapabilityReceiptBody } | { ok: false, error: VerifyError }`
- `VerifyError` discriminated by `kind: "key-not-found" | "key-revoked" | "canonicalization-mismatch" | "signature-invalid" | "envelope-malformed" | "version-mismatch"`. Typed verify error union is partially implemented in v1.1; the `EnvironmentDrift` / `RedactionDrift` / `Tampered` variants come in v1.2.
- Verification re-canonicalizes the body from the signature payload, re-computes PAE, and verifies with WebCrypto. NO trust in any embedded fields without verifying them.

### Emission Strategy
- Receipts emitted whenever `LatticeConfig.signer` is configured, regardless of contract presence. `contractVerdict: "success"` always emitted on success runs even when no contract was attached (records the run for compliance).
- On no-contract-match: receipt carries `noRouteReasons[]`. `outputHash` is null. `usage` is zero.
- On tripwire-violated: receipt carries `tripwireEvidence`. `outputHash` is null. `usage` reflects cost-so-far.
- On execution-failed: receipt records the failure kind in `contractVerdict`. `outputHash` is null. `usage` reflects whatever was spent.
- On validation-failed: same as execution-failed; outputs that failed schema validation are NOT hashed (no commitment to invalid output).
- Receipts attach to `RunSuccess.receipt?` and `RunFailure.receipt?` (NEW optional field on each result variant).

### Claude's Discretion
- Exact file layout under `packages/lattice/src/receipts/` is at Claude's discretion. Recommended: `receipt.ts` (createReceipt + types), `verify.ts` (verifyReceipt), `envelope.ts` (DSSE PAE + base64 helpers), `sign.ts` (WebCrypto Ed25519 wrappers + createInMemorySigner), `canonical.ts` (canonicalize wrapper + costUsd-to-string), `redact.ts` (receipt-specific redactor), `keyset.ts` (KeySet interface and createMemoryKeySet helper).
- UUID generator implementation (use `crypto.randomUUID()` from Node 24 — already available).
- Internal layout of `redactions[]` paths (dotted vs JSON Pointer): use dotted paths for consistency with the Phase 8 tripwire path resolver.
- Whether to expose `createReceiptBody` (the pre-sign builder) publicly: keep internal in v1.1, may go public in v1.2 for custom signers.
</decisions>

<specifics>
## Specific Ideas

- The DSSE-shaped envelope is just a serialization concern. Internal types are TypeScript; the envelope is what's persisted/transmitted. Round-trip helpers (`envelope.encode(receipt) -> ReceiptEnvelope` and `envelope.decode(envelope) -> { payloadType, payloadBytes, signatures }`) keep PAE construction in one place.
- RFC 8785 appendix vectors MUST be in the test suite as known-good canonicalization inputs. If `canonicalize@3.0.0` deviates from RFC 8785 for any vector, fail the test. This catches version-pinning drift early.
- `@noble/ed25519@3.1.0` parity oracle: in a single test file, sign a known input with WebCrypto, sign the same input with noble, assert the signatures byte-equal. Run on every CI build. This catches WebCrypto regressions silently introduced by Node updates.
- `costUsd` serialization: when building the canonical receipt body, convert `number | null` to `string | null` ("0.000125" rather than `0.000125`). This avoids float drift across V8 / Bun / Deno engines. The `usage` field on the public `RunResult` keeps the number — only the receipt-internal representation changes.
- Performance: the receipt builder must complete in <5ms for a typical run on Node 24. Property-based test: 100 random runs each issue a receipt in <50ms total.
</specifics>

<deferred>
## Deferred Ideas

- Typed verify error union extensions: `EnvironmentDrift`, `RedactionDrift`, `Tampered` (deferred to v1.2).
- Lineage merkle root in signed payload (deferred to v1.2).
- Signed redaction-policy registry (`redactionPolicyId` is a free-form string in v1.1).
- KMS adapter shapes (AWS KMS / GCP KMS / OS keyring).
- `lattice receipt diff` subcommand.
- Per-run signer overrides on `ai.run` (signer is `LatticeConfig`-level only in v1.1).
- Cross-Node version JCS appendix vector matrix CI (single-version verification in v1.1).
- Receipt-aware verifier MCP tool.
</deferred>
