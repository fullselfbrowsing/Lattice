# Plan 15-01: Capability Receipt v1.1 schema extension (step-marker fields)

**Status:** Complete (retro).
**Wave:** 1
**REQ-IDs:** RECEIPT-EXT-01, RECEIPT-EXT-02, RECEIPT-EXT-03
**Originating commit:** `5c48134` feat(receipts): extend CapabilityReceiptBody with step-marker fields + bump to v1.1.

## Goal

Widen `CapabilityReceiptBody.version` from the v1.1 literal `"lattice-receipt/v1"` to the literal union `"lattice-receipt/v1" | "lattice-receipt/v1.1"`, add six optional step-marker fields, and teach `createReceipt` to auto-bump emitted receipts to `v1.1` when any step-marker field is populated.

## Tasks

1. `packages/lattice/src/receipts/types.ts` — widen `CapabilityReceiptBody.version` to the literal union. Add 6 optional fields: `stepName?: string`, `stepIndex?: number`, `parentStepName?: string`, `previousStepName?: string`, `sessionId?: string`, `timestamp?: string` (ISO-8601). Alphabetical order preserved for JCS canonicalization.
2. `packages/lattice/src/receipts/receipt.ts` — `createReceipt` calls a new internal `hasStepMarker(input)` helper; when truthy, the emitted body's `version` is set to `"lattice-receipt/v1.1"`. Otherwise `v1` continues to be emitted.
3. `packages/lattice/src/receipts/verify.ts` — verifier accepts either literal in the body's `version` field. No structural change to DSSE/PAE.
4. New tests: `receipt.test.ts` covers v1.1 mint with all 6 fields, partial step-marker mint (v1.1 trigger), no-step-marker mint (still v1). `verify.test.ts` covers verify against both v1 and v1.1 receipts.

## Acceptance

- v1.0 callers compile and run unchanged. Existing on-disk v1 receipts verify.
- Emitting a receipt with any step-marker field populated produces a body whose `version === "lattice-receipt/v1.1"`.
- JCS canonicalization round-trip preserved for v1.1 receipts (Phase 15 byte-equal property test).

Lattice vitest gains 6 receipt tests + 2 verify tests = 8 cases.
