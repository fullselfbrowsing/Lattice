# Plan 19-03: formatToolsForProvider helper + native tool_use parse

**Status:** Not started
**Wave:** 2 (depends on 19-02 type foundation)
**REQ-IDs:** AGENT-02

## Goal

Ship a provider-aware helper that formats Lattice's v1.0 `Tool[]` into each provider's native tool_use shape, plus a parser closure that extracts tool-call requests from provider responses. Keeps `ProviderAdapter` shape unchanged (INV-03 7-provider parity preserved).

## Tasks

### Task 1: `packages/lattice/src/agent/format-tools.ts`

Export `formatToolsForProvider(providerName, tools, mode?)`:

- `providerName: string` â€” e.g. `"openai"`, `"openai-compat"`, `"anthropic"`, `"gemini"`, `"xai"`, `"openrouter"`, `"lm-studio"`.
- `tools: readonly Tool[]` â€” v1.0 tool primitives.
- `mode?: "native" | "prompt-reencoded" | "auto"` (default `"auto"`).

Returns `{ requestFragment: Record<string, unknown>; parseToolUse: (response: ProviderResponse) => ReadonlyArray<ToolUseRequest> | null }`.

`ToolUseRequest` â€” `{ id: string; name: string; args: unknown }`.

Per provider:

| Provider | Native shape | Parse path |
|---|---|---|
| `openai`, `openai-compat`, `xai`, `openrouter` (passthrough) | `tools: [{ type: "function", function: { name, description, parameters } }]` | `response.choices[0].message.tool_calls[]` |
| `anthropic` | `tools: [{ name, description, input_schema }]` | `response.content[]` filtered to blocks where `type === "tool_use"` |
| `gemini` | `tools: [{ function_declarations: [...] }]` | `response.candidates[0].content.parts[]` filtered to `functionCall` parts |
| `lm-studio` (auto fallback) | Prompt-reencode tools into a system+user prompt block | JSON-parse final assistant message for a tool-call envelope |

Schema translation: each `Tool.schema` (Standard Schema) is converted to JSON Schema via a small reusable helper `toolSchemaToJsonSchema(schema)`. For Standard Schema implementations that expose `~standard.vendor`, prefer the vendor's `toJSONSchema` if available; otherwise use a structural conversion (object/string/number primitives + nested objects).

### Task 2: Tool-use response parsing

For each native protocol, the `parseToolUse` closure returns:
- `null` â€” response is a final assistant answer (no tool_use blocks).
- `ToolUseRequest[]` â€” one or more tool calls to dispatch.

`response: ProviderResponse` is whatever the adapter returned; the closure has provider-specific knowledge of how to extract the tool calls. No coupling: `ProviderAdapter` interface unchanged.

### Task 3: Prompt-reencoded fallback

`mode: "prompt-reencoded"` (or `auto` when target is `lm-studio`):
- Build a system prompt block describing the tools: name, description, JSON-Schema-encoded args.
- Instruct the model to either answer directly OR output `{ "tool_call": { "name": "...", "args": {...} } }` as a JSON object on a line by itself.
- `parseToolUse` regex-extracts JSON objects matching the envelope shape; returns `null` if absent.
