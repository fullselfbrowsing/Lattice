## Accomplishments

- `lattice repro <target>` honours three resolution paths for sidecars (explicit --sidecar, --sidecar-dir/<id>.json, convention <receiptsDir>/../sidecars/<id>.json) in strict precedence order. Stage 3.5 inserts the lookup BEFORE the Stage-4 materializer, so the sidecar's `{ task, outputs, policy, contract }` quadruple is spread directly into `materializeReplayEnvelope`. With a sidecar present, replay succeeds and the outputHash comparison runs.
- Convention-miss with no flag is non-fatal. The Stage-6 replay-failed branch appends `hint: Provide --sidecar <path> or place a sidecar at .lattice/sidecars/<receipt-id>.json. See lattice-sidecar/v1 spec.` to stderr so users can copy-paste a path. Exit code matrix unchanged.
- Explicit-flag sidecar load failures surface as `FAIL kind=sidecar-load-failed reason=<original-kind> at <path>: <detail>`. Every `SidecarLoadError` discriminator from Plan 01 (file-not-found / malformed / version-mismatch / unsupported-output-shape) maps cleanly; version-mismatch and unsupported-output-shape additionally include `received=` and `outputKey=` so users can fix the sidecar without re-reading the source.
- `lattice eval` swaps `walkReceiptsDirectory` for `walkReceiptsWithSidecars`. Per-fixture sidecars (when present) apply through `applySidecar(...)` into the materializer. Fixtures without a sidecar are explicitly tagged `loadFailedReason: "no-sidecar"` instead of being silently silenced — the JSON projection now distinguishes "audit can't decide" from "no input quadruple was provided".
- `FixtureReport.loadFailedReason` is the new typed discriminator. Every load-failed branch in the runner threads the right reason (`verify-failed` / `replay-failed` / `malformed-sidecar` / `outputhash-missing` / `no-sidecar`); every non-load-failed branch sets it to `null`. Additive on `lattice-eval/v1` — pre-Plan-13.1 consumers ignore unknown fields.
- The Stage-8 cost-regression gate now actually fires for fixtures with a usable sidecar (V1.1-LIMITATION-2 closure). Test 15 in `eval-runner.test.ts` is the audit's "unreachable" case made reachable: sidecar present, replay outputs hash-match, baseline cost lower than receipt cost → verdict=regression, regressionKind=cost-regression.
- `EvalConfig.sidecarsDir` is a first-class field defaulting to `.lattice/sidecars`. `lattice eval --sidecar-dir <dir>` exposes it on citty's CLI surface.

## FixtureReport.loadFailedReason Taxonomy

| Value                  | When emitted                                                                                                                                  |
| ---------------------- | --------------------------------------------------------------------------------------------------------------------------------------------- |
| `null`                 | Every non-load-failed verdict (match / drift / regression).                                                                                   |
| `"no-sidecar"`         | Walker yielded the receipt but `entry.sidecar === null`. The Plan 13.1 explicit case.                                                          |
| `"malformed-sidecar"`  | Walker surfaced a sidecar-side load error (malformed / version-mismatch / unsupported-output-shape). Disambiguated by sidecarsDir prefix match. |
| `"verify-failed"`      | Receipt-side walker error OR materializeReplayEnvelope threw. Pre-Plan-13.1 generic load-failed cases land here.                              |
| `"replay-failed"`      | replayOffline returned `ok: false` (typically execution_unavailable, now uncommon since the sidecar threads outputs).                          |
| `"outputhash-missing"` | Verified body has `outputHash === null` (failure / tripwire receipts). Cannot diff; surfaced for the audit.                                    |

## Decisions Made

- **Receipt id derivation chosen BEFORE Stage 5 re-verify.** The convention sidecar lookup must not depend on a verified body — Stage 5's `verifyReceipt` is microsecond-level but conceptually it's downstream of Stage 4's materialize. Deriving the id from the on-disk filename (`basename(loaded.resolvedPath, ".json")`) preserves the existing pipeline order and matches the `writeReceipt` convention. For bare-id targets the id is used verbatim.
- **WalkedReceiptError single-union preserved.** Sidecar-side errors keep the `kind: "malformed"` shape from Plan 01's walker; the runner disambiguates via `resolvedPath.startsWith(resolvedSidecarsDir)` rather than forcing a wider union onto consumers. This keeps the consumer surface from Plan 01 unchanged and lets `loadFailedReason` carry the precise sub-discriminator.
- **No version bump on `lattice-eval/v1`.** The new `loadFailedReason` field is additive. Pre-Plan-13.1 readers ignore unknown fields; new readers can switch on the discriminator. This matches the v1.1.1 sub-phase decision in `13.1-CONTEXT.md` and avoids forcing every downstream tool to bump.
- **Convention-miss is non-fatal even with a success receipt.** Originally the plan said "fall back to today's behavior (returns `execution_unavailable`)". After running the test matrix, the right exit code is still 2 (replay-failed) — what changes is the addition of a one-line stderr hint pointing at the convention. Users get actionable advice; CI scripts get the same exit code.
- **`seedFixtureOnDisk` writes a default sidecar.** With the runner now expecting sidecars, every pre-Plan-13.1 eval-runner test would have flipped to `load-failed` without one. Writing a valid default sidecar alongside each fixture (with an `skipSidecar` opt-out) keeps the 11 pre-existing test cases green and minimises churn. The 4 new test cases override the helper to exercise the no-sidecar / malformed-sidecar branches.
