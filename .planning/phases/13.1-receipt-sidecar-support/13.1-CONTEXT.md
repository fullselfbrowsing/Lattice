# Phase 13.1: Receipt Sidecar Support - Context

**Gathered:** 2026-05-12
**Status:** Ready for planning

<domain>
## Phase Boundary

Extend `lattice repro` and `lattice eval` to accept a `--sidecar <path>` flag pointing to a JSON file containing the original `RunIntent` inputs (`task`, `outputs`, `policy`, `contract`) that the receipt body intentionally does not embed. This closes the documented v1.1 architectural limitation that blocked end-to-end replay round-trips and the eval regression gate from actually firing.

Out of scope: receipt body schema changes (receipts stay minimal); CONTRACT/RECEIPT REQ-IDs (they are already satisfied — Phase 13.1 only unblocks CLI-02 and EVAL-02/EVAL-06 end-to-end paths).
</domain>

<decisions>
## Implementation Decisions

### Sidecar File Format
- JSON file at `.lattice/sidecars/<receipt-id>.json` (or arbitrary path via flag).
- Schema:
```json
{
  "version": "lattice-sidecar/v1",
  "task": "string",
  "outputs": { "<name>": "<output-contract-spec>" },
  "policy": { ... PolicySpec ... },
  "contract": { ... CapabilityContract ... }
}
```
- `outputs` cannot be serialized losslessly when it contains StandardSchema/Zod validators (those are functions). v1.1 sidecars carry primitive-serializable subsets:
  - For `"text"` literal outputs: serialized as `"text"`.
  - For Standard Schema validators: serialized as `{ "type": "standard-schema-placeholder", "shape": "<best-effort JSON schema sketch>" }`. At load time, the sidecar loader rehydrates only the literal types; non-trivial schemas surface a clear loader error pointing users at v1.2.
- `contract` is fully serializable (already JSON-canonicalizable per Phase 9 hash work).
- `policy` is fully serializable.

### Loader Behavior
- New module: `packages/lattice-cli/src/io/sidecar-loader.ts` exporting `loadSidecar(path: string): Promise<SidecarFile>` and `applySidecar(receipt, sidecar): Partial<RunIntent>`.
- The loader is strict: malformed JSON → `SidecarLoadError` discriminated by kind `"file-not-found" | "malformed" | "version-mismatch" | "unsupported-output-shape"`.
- Convention resolution: if `--sidecar` is omitted, the CLI checks `<receiptsDir>/../sidecars/<receipt-id>.json` automatically. Missing-sidecar with no flag is non-fatal: `lattice repro` falls back to the v1.1 behavior (returns `execution_unavailable`) with a helpful error message pointing at the convention.

### `lattice repro` Update
- New flag: `--sidecar <path>` and optional `--sidecar-dir <dir>` (default `.lattice/sidecars/`).
- When a sidecar is found, the materializer is called with `{ task, outputs, policy, contract }` from the sidecar passed into `materializeReplayEnvelope`'s existing optional parameters (Phase 10 already accepts these).
- Replay then succeeds and the outputHash diff actually runs.
- Exit codes unchanged (0 match, 1 drift, 2 verify/materialize/load failure).

### `lattice eval` Update
- Reuse the same sidecar loader.
- Walker now pairs each receipt with its sidecar (when present); fixtures without sidecars are marked `verdict: "load-failed"` with `loadFailedReason: "no-sidecar"` instead of being silently silenced.
- Configuration: `--sidecar-dir <dir>` (default `.lattice/sidecars/`).
- Regression gate now actually fires because fixtures can reach `verdict: "match"` and cost/quality comparisons run.

### Showcase Update
- `examples/work-inbox/setup.mjs` adds `writeSidecar(receiptId, intent)` helper.
- Each scenario writes its sidecar to `examples/work-inbox/.lattice/sidecars/<receipt-id>.json` after the receipt.
- Stdout next-steps update to reference the sidecar locations.

### showcase-e2e Test Update
- Case 3 (`lattice repro`) now expects exit 0 with `verdict=match` (the documented forward-compat branch flips).
- Case 6 (`lattice eval` regression flip) now actually fires exit 1 (the conditional branch resolves).
- Update the `--known-limitation` annotations in 13-02-SUMMARY.md to mark these two as resolved by Phase 13.1.

### Claude's Discretion
- Whether to alias `--sidecar` and `--sidecar-dir` consistently across both subcommands (recommend yes; identical names and defaults reduce mental load).
- Internal module layout for the sidecar walker pairing (recommend: `packages/lattice-cli/src/io/sidecar-walker.ts` that mirrors `receipt-walker.ts`'s async iterable shape).
- Whether to error or warn when a sidecar exists but its `task` is empty: warn (informational) — eval can still gate on cost; only `lattice repro` requires a non-empty task.

### Limitations
- StandardSchema/Zod validators in `outputs` cannot be fully serialized. v1.1.1 (this sub-phase) supports literal outputs (`"text"`) and the citation/artifact contract types. Generic Zod schemas defer to v1.2 with a richer schema-bridge.
- No backwards-compatible auto-detection of "old" sidecars — the `version` field is required from day one.
</decisions>

<specifics>
## Specific Ideas

- The sidecar file convention should mirror the receipt convention exactly: `<id>.json` filename, content-addressed by the receipt's `receiptId` field. This way `.lattice/receipts/abc.json` pairs with `.lattice/sidecars/abc.json` automatically.
- The walker pairing logic is trivial: for each receipt yielded by `walkReceiptsDirectory`, attempt to load the matching sidecar; emit `{ id, envelope, sidecar }` on success or `{ id, envelope, sidecar: null }` if missing.
- The showcase should print exact next-step commands like: `node packages/lattice-cli/dist/cli.js repro <receipt-id> --fixtures examples/work-inbox/.lattice/fixtures --sidecar-dir examples/work-inbox/.lattice/sidecars --key examples/work-inbox/.lattice/keyset.json` so a user copy-pastes and sees the full round-trip work.
</specifics>

<deferred>
## Deferred Ideas

- Full StandardSchema/Zod serialization bridge (deferred to v1.2 — Phase 13.1 supports literal `"text"` outputs and the built-in citation/artifact contracts).
- Sidecar signing (sidecars themselves are NOT signed in v1.1.1; the receipt is the trust anchor; sidecars are inputs to replay, not attestations). v1.2 may sign sidecars.
- Multi-sidecar bundles (one JSON containing many sidecars) — deferred.
- `lattice sidecar diff` subcommand — deferred.
- Generating sidecars retroactively from running runtime instrumentation — deferred.
</deferred>
