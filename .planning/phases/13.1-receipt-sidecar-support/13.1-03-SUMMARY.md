## Sidecar Wire-Format Evolution

The v1.1 sidecar file format gains ONE optional field. The existing v1 surface is unchanged.

```json
{
  "version": "lattice-sidecar/v1",
  "task": "...",
  "outputs": {
    "answer": "text",
    "evidence": { "kind": "citations" },
    "generated": { "kind": "artifacts" }
  },
  "policy": { "privacy": "sensitive" },
  "contract": { "kind": "capability-contract", "budget": { "maxCostUsd": 0.05 } },
  "rawOutputs": {
    "answer": "Approve a replacement and escalate billing review...",
    "evidence": [{ "artifactId": "artifact:text:message", "label": "customer message" }],
    "generated": []
  }
}
```

| Field        | Required | Purpose                                                                                                                              |
| ------------ | -------- | ------------------------------------------------------------------------------------------------------------------------------------ |
| `version`    | yes      | Strict version check; only `"lattice-sidecar/v1"` accepted.                                                                          |
| `task`       | yes      | Original `RunIntent.task` string.                                                                                                    |
| `outputs`    | yes      | Schema spec — `"text"` / `{ kind: "citations" }` / `{ kind: "artifacts" }` only. Standard Schema validators still surface `unsupported-output-shape`. |
| `policy`     | yes      | Original `RunIntent.policy`.                                                                                                         |
| `contract`   | yes      | Original `RunIntent.contract`.                                                                                                       |
| `rawOutputs` | **no**   | Actual validated provider output VALUES. When present, `applySidecar` projects these into the materializer's `outputs` so `lattice repro` recomputes the receipt's outputHash → verdict=match. When omitted, the schema spec is rehydrated and replay produces a non-matching hash (acceptable for fixtures whose receipts have `outputHash === null`). |

## Decisions Made

See `key-decisions` in frontmatter. Most consequential:

1. **`rawOutputs` is additive, not a v2 schema.** The Plan 01 loader's `version-mismatch` discriminator was deliberately strict from day one. Adding `rawOutputs?: Record<string, unknown>` as an optional v1 field (with a tight `isPlainObject` shape check) preserves the strict-from-day-one promise while unblocking the plan's `verdict=match` success criterion. Existing v1 sidecars without the field still load; new tooling that needs values opts in by populating it.
2. **Failure-class scenarios do NOT carry rawOutputs.** Tripwire + no-contract-match receipts have `outputHash === null` by design — failure receipts cannot commit to outputs because no validated output set exists. Their sidecars carry only the schema spec so they remain discoverable by the eval walker (avoiding `loadFailedReason="no-sidecar"`), but `lattice repro` and the eval comparator still classify them as `load-failed, loadFailedReason="outputhash-missing"` — the correct documented behavior.
3. **No envelope-version bump on showcase-e2e's EvalReport type.** The test's local `EvalReport` interface only adds the additive `loadFailedReason: string | null` field that Plan 02 already documented as additive on `lattice-eval/v1`. Test code mirrors the runtime additive contract.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 2 - Missing Critical] Sidecar's v1 wire format could not reach verdict=match**
- **Found during:** Task 1 first real `lattice repro` invocation against the showcase output.
- **Issue:** Plan 03 (and Plan 01's CONTEXT.md) specified that the sidecar's `outputs` field carries schema spec literals (`"text"` / `{ kind: "citations" }` / `{ kind: "artifacts" }`). `applySidecar` rehydrates these into `OutputContractMap` values and the materializer sets `envelope.outputs = options.outputs`. But the receipt's recorded `outputHash` is `sha256(JSON.stringify(provider_validated_outputs))` — i.e. the actual VALUES, not the schema spec. `lattice repro` then computes `sha256(JSON.stringify(schema_map))` which can NEVER match. The plan's success criterion ("HARD assertion: exit 0 with verdict=match") was architecturally impossible with the as-shipped sidecar format. The Plan 02 eval-runner tests passed only because they mock `replayOffline` to return the original outputs.
- **Fix:** Add an OPTIONAL additive field `rawOutputs?: Record<string, unknown>` to the v1 wire format. Loader validates plain-object shape; `applySidecar` prefers `rawOutputs` (values) over the schema spec when present. The success scenario writes `result.outputs` (validated provider values from `ai.run`) into `rawOutputs`. Real `lattice repro` now produces the same outputHash as the receipt.
- **Files modified:** `packages/lattice-cli/src/io/sidecar-loader.ts`, `examples/work-inbox/scenarios/success.mjs`
- **Verification:** `node packages/lattice-cli/dist/cli.js repro <success-id> --sidecar-dir <dir> ...` exits 0 with stdout `verdict=match`. `pnpm vitest run` 141/141 green; the existing Plan 01 sidecar-loader tests (which never populate `rawOutputs`) still pass unchanged because the field is optional.
- **Committed in:** `4e5ab51` (Task 1 commit)
- **Rule:** Rule 2 (missing critical functionality required for the plan's stated outcome). The plan author intended verdict=match to work; they just didn't trace the hash path back to provider values. Additive schema evolution, no breaking change, no architectural redesign — qualifies as Rule 2, not Rule 4.
