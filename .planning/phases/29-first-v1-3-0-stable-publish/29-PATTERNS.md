## Existing Analogs

### Release Workflow Shape

Analog: `.github/workflows/release.yml`

Pattern:

- Root permissions stay `contents: read`.
- `version-pr` has `contents: write` and `pull-requests: write`, but no `id-token`.
- `publish` has `id-token: write`, uses `environment: npm-publish`, and creates GitHub Release after npm publish.
- Third-party actions are SHA-pinned.
- No `NODE_AUTH_TOKEN` or `NPM_TOKEN`.

Planner guidance:

- Any workflow edit must preserve this shape exactly.
- If adding changelog-derived release notes, add a step immediately before `Create GitHub Release` and pass the generated file/body to the existing `softprops/action-gh-release` step.
- Executor must verify the pinned `softprops/action-gh-release` input name (`body_path` or equivalent) against the action source/docs before editing.

### CI and Static Release Gates

Analog: `.github/workflows/ci.yml`

Pattern:

- Sequential gates: install -> build -> typecheck -> test -> test:types -> lint packages -> tarball audit -> rename audit -> workflow safety audit.
- Workflow safety has a dedicated script rather than relying on visual YAML inspection.

## Data Flow

1. Source changes + pending changesets land on `main`.
2. `version-pr` runs `pnpm changeset version`, mutating manifests and changelogs in a Version Packages PR.
3. User reviews/merges Version Packages PR.
4. Changesets tag `v1.3.0` triggers `publish`.
5. `publish` builds/lints, publishes both packages through npm OIDC, then creates GitHub Release.
6. Postflight commands query npm and GitHub and write proof into Phase 29 summary/status artifacts.

## Risks to Preserve in Plans

- Keep external UI handoffs non-autonomous.
- Never add token fallback.
- Never run full publish retry after partial package publish without a recovery plan.
- Do not let generic GitHub generated notes replace package changelog content for stable.
