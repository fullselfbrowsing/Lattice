### Documented v1.1 Limitations

**RESOLVED in this re-audit:**

1. **V1.1-LIMITATION-1** (`lattice repro` returns `execution_unavailable` against a receipt-only envelope) — **RESOLVED by Phase 13.1.** The sidecar's optional additive `rawOutputs` field carries the original provider output values; `applySidecar` projects them into `materializeReplayEnvelope`; the replay's recomputed outputHash matches the receipt's committed hash; `lattice repro` exits 0 with `verdict=match`. Empirically confirmed by direct CLI invocation and showcase-e2e case 3 HARD assertion.

2. **V1.1-LIMITATION-2** (`lattice eval` cost-regression gate cannot fire in v1.1) — **RESOLVED by Phase 13.1.** Same root cause as #1: every showcase fixture now reaches stage 8 (cost comparator) once paired with a sidecar. Empirically confirmed: mutated baseline with `costUsd=-0.0001` for the success fixture -> exit 1, JSON report shows `verdict=regression, regressionKind=cost-regression, deltaCostPct=-1`. Showcase-e2e case 6 HARD-asserts this on every CI run.

## Cross-Cutting Verification

- All `lattice` public exports referenced from `lattice-cli` resolve to the package surface in `packages/lattice/src/index.ts` (verified by inspecting every `from "lattice"` import in `packages/lattice-cli/src`).
- Sidecar primitives (loader, walker, applySidecar) live in `packages/lattice-cli/src/io/` and are NOT re-exported from the lattice public surface — CLI-06 boundary preserved post-13.1.
- `Usage` shape consistency across phases:
  - Phase 7 introduces `Usage { promptTokens: number; completionTokens: number; costUsd: number | null }` in `providers/provider.ts`
  - Phase 9 serializes to `ReceiptUsageCanonical { promptTokens: number; completionTokens: number; costUsd: string | null }` for I-JSON compliance (no raw floats)
  - Phase 12 reads `body.usage.costUsd ?? "0"` in `runner.ts` `usageFromBody` and `compareCost` — string normalization is intentional, lossy boundary documented
  - Phase 13.2 surfaces the normalized shape across all three adapter families (openai catalog-priced, openai-compat caller-priced, ai-sdk null-cost) via the `adapterUsage` stdout lines
- `isTerminal` exported from Phase 8 (`results/errors.ts`) and consulted by runtime fallback short-circuit at `runtime/create-ai.ts:427`.
- `verifyReceipt` is invoked by FOUR independent code paths: (a) `lattice verify` command, (b) `lattice repro` command, (c) `lattice eval` runner (Stage 3 typed-body fetch), (d) `materializeReplayEnvelope` (Step 1 verify-FIRST). Cross-phase fan-out is broad and tested.
- `KeySet` loaded once from disk via `loadKeySetFromPath` (Phase 11) and reused across verify / repro / eval / showcase entry points.
- Sidecar precedence chain in repro: `--sidecar` > `--sidecar-dir/<id>.json` > `<receiptsDir>/../sidecars/<id>.json` (convention). Convention-miss without explicit flag is non-fatal (stderr hint, exit 2). Explicit-flag failures surface as `FAIL kind=sidecar-load-failed reason=<original-kind>` with one of four discriminated kinds.
- `FixtureReport.loadFailedReason` additive on `lattice-eval/v1` (no envelope-version bump); six-value typed union (null | no-sidecar | malformed-sidecar | verify-failed | replay-failed | outputhash-missing). Failure-class receipts (tripwire / no-contract-match) correctly classify as `loadFailedReason="outputhash-missing"` because `body.outputHash === null` by design.
- `body.contractHash` cross-phase observability: showcase quality-floor scenario locally re-computes `sha256(canonicalize(contract))` via direct relative import of canonicalize@3.0.0; showcase-e2e CONTRACT-03 case reads the same field from the base64url-decoded payload of the on-disk DSSE envelope; equality is the observable canonicalization proof.
