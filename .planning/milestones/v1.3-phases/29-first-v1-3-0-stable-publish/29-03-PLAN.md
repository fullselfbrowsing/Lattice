  <action>
    Monitor the release workflow until terminal state:
    ```bash
    gh run watch <run-id> --repo fullselfbrowsing/Lattice --exit-status
    ```
    If the run succeeds, record success and continue to Plan 04.
    If the run fails, immediately query:
    ```bash
    npm view @full-self-browsing/lattice versions --json
    npm view @full-self-browsing/lattice-cli versions --json
    gh release view v1.3.0 --repo fullselfbrowsing/Lattice --json tagName,url 2>/dev/null || true
    ```
    Apply this recovery tree exactly:
    - neither package has `1.3.0`: preserve run URL/log summary, diagnose, then rerun only after fix;
    - exactly one package has `1.3.0`: stop and create a recovery plan for targeted OIDC publish of the missing package or a `1.3.1` bump;
    - both packages have `1.3.0` but GitHub Release failed: continue to a GitHub Release repair task in Plan 04;
    - any package published without provenance/attestations: stop and mark Phase 29 failed.
  </action>
  <verify>
    <manual>Record workflow terminal state and, on failure, record npm side-effect classification before any rerun.</manual>
  </verify>

    - No rerun occurred after partial publish without a written recovery plan.
  </acceptance_criteria>
  <done>Stable publish side effect is either complete or safely checkpointed for recovery.</done>
