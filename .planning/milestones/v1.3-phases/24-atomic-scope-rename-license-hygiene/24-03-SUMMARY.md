## showcase-e2e.test.ts line 204 disposition (REQUIRED documentation)

**Decision:** REWRITTEN to `"@fullselfbrowsing/lattice"` (package-name reference, NOT bin-name reference).

**Justification — surrounding code (lines 198-220 post-rewrite):**

```ts
// Build the runtime first — the showcase imports the lattice dist.
// `pnpm --filter lattice-cli test` only triggers the CLI build via the
// package.json `test` script; it does NOT transitively build the
// runtime, so we do it explicitly here.
const latticeBuild = await runProc("pnpm", [
  "--filter",
  "@fullselfbrowsing/lattice",      // <-- line 204
  "build",
]);
...
const cliBuild = await runProc("pnpm", [
  "--filter",
  "@fullselfbrowsing/lattice-cli",  // <-- line 216
  "build",
]);
```

The literal at line 204 is the second element of the `args` array passed to `runProc("pnpm", [...])`, immediately following `"--filter"` and immediately preceding `"build"`. `pnpm --filter <name>` matches its `<name>` argument against the workspace's `package.json#name` field. After Plan 24-01's name flip, the workspace member is named `@fullselfbrowsing/lattice` — so `pnpm --filter lattice build` would now match nothing and the test would fail with "No projects matched the filters." The rewrite to `@fullselfbrowsing/lattice` is REQUIRED for the test to keep building the runtime.

Line 216 follows the same pattern for the CLI build step and was rewritten to `@fullselfbrowsing/lattice-cli` for the identical reason.

Neither literal is a bin-name reference. Bin-name references in this codebase show up as `spawn(CLI_BIN, [...])` or `runProc(CLI_BIN, [...])` where `CLI_BIN` is the resolved binary path; they never appear as a string literal `"lattice"` directly in a `runProc` argv. The `bin.lattice` entry in `packages/lattice-cli/package.json` is preserved (verified by the tarball inspection — `cli.bin.lattice === "./dist/cli.js"`).

## Deviations from Plan

This plan executed under a continuation hand-off after the prior executor reached a checkpoint. Three deviations are documented:

### 1. Pre-existing typecheck errors fixed in a SEPARATE prep commit before the atomic rename (user-approved)

When the prior executor reached the typecheck gate (Task 2 Step 2), `pnpm -r typecheck` reported three pre-existing failures in `packages/lattice/test/agent/*.test.ts` where `recoveryEvents` payloads violated `exactOptionalPropertyTypes`. These errors were NOT introduced by the rename — they existed on `HEAD` before Phase 24 started. The prior executor checkpointed with three options for the user:

- Option 1: amend the rename commit with the type fix (rejected — logically distinct concerns)
- Option 2: skip typecheck gate temporarily and let a follow-up fix it (rejected — undermines the verification matrix)
- Option 3: land a separate prep commit on `HEAD` BEFORE the rename atomic commit (selected)

The orchestrator landed Option 3 as commit `bbcb4ac fix(types): widen recoveryEvents payload type to satisfy exactOptionalPropertyTypes`, and this plan's continuation picked up with `HEAD == bbcb4ac` and the 22 Phase 24 files still staged. The atomic rename commit `a267048` was then produced on top of `bbcb4ac`. This deviation is structurally clean: the rename commit is still atomic for all Phase 24 file changes, and the type fix sits as a logically separate predecessor commit that future history-archaeology can interpret cleanly (`git log a267048^..a267048` shows only Phase 24 changes; `git log bbcb4ac^..bbcb4ac` shows only the type widening).

**Disposition:** [Rule 4 — Architectural decision] User-approved before this continuation began. Documented here for traceability.

### 2. examples/agent-loop/package.json included in the atomic commit (scope expansion documented but matching the plan)

The original 24-01 plan focused on `packages/lattice` and the workspace root. The 24-02 plan focused on `packages/lattice-cli`. The 24-03 plan's Task 1 edit F explicitly names `examples/agent-loop/package.json` for the dep flip. So this is technically WITHIN the plan, but worth highlighting because it expanded the file-list beyond what 24-01 + 24-02 staged — the prior executor correctly added it during Task 1 (the staged tree at start of this continuation already included it).

The rationale (matches PITFALLS RENAME-1): `examples/agent-loop` is `"private": true` and never publishes, but it consumes `@fullselfbrowsing/lattice` as a workspace dep. If the dep key stayed unscoped, `pnpm install` would either fail to resolve or silently fall through to npm (potentially pulling a squatted `lattice` package once one exists). The flip to `@fullselfbrowsing/lattice: workspace:^` aligns it with `packages/lattice-cli` and ensures the example always resolves to the local workspace member.

**Disposition:** No deviation — explicitly in the plan. Highlighted for the orchestrator's audit because it's the only `examples/` file in the commit and could be mistaken for scope creep.

### 3. showcase-e2e.test.ts pnpm-filter args rewritten (lines 204, 216)

The plan's Task 1 edit E flagged `showcase-e2e.test.ts` line ~204 as "context-dependent" — could be a bin-name reference (keep as `"lattice"`) or a package-import reference (rewrite). Both line 204 (`"@fullselfbrowsing/lattice"`) and the matching line 216 (`"@fullselfbrowsing/lattice-cli"`) were rewritten.
