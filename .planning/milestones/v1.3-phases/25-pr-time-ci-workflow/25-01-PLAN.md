<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| workspace -> tarball | publishable package.json fields cross into the published artifact at `pnpm pack` time |
| workspace -> CI runner | gate scripts execute on the GitHub-hosted ubuntu-latest runner with no secrets and no OIDC token (ci.yml scope only) |
| workspace -> attacker (post-publish) | unscoped `lattice` references shipped in the registry tarball would let an unscoped-name squatter steer downstream consumers via a typosquat manifest |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-25-01 | Tampering | scripts/check-tarball-leak.mjs | mitigate | hard-codes the publishable package set; no dynamic discovery so an attacker adding a new package directory cannot bypass the gate |
| T-25-02 | Information Disclosure | scripts/check-tarball-leak.mjs | accept | tmpdir cleanup runs in `finally`; transient tarballs live only in OS temp space, ubuntu-latest runners are ephemeral |
| T-25-03 | Tampering | scripts/verify-rename.mjs | mitigate | ALLOWLIST_FILES literal is the only escape hatch; adding to it is a code review event |
| T-25-04 | Denial of Service | scripts/verify-rename.mjs | mitigate | SKIP_NAMES set excludes node_modules + .git + dist + .planning so the walk completes in seconds even on a populated workspace |
| T-25-05 | Elevation of Privilege | scripts/check-workflow-safety.mjs | mitigate | implements the TanStack blast-radius defense (OIDC-1): no non-publish job may carry `id-token: write`; no workflow may use `pull_request_target` |
| T-25-06 | Spoofing | scripts/check-workflow-safety.mjs | mitigate | release.yml allowance is hard-coded to basename equality; a workflow file renamed to look like `release.yml` in a non-canonical path would still need a `publish` job key, requiring two simultaneous evasions |
| T-25-07 | Repudiation | all three scripts | accept | single-line greppable log output is sufficient evidence for post-hoc review; CI logs are retained by GitHub Actions per repo policy |
