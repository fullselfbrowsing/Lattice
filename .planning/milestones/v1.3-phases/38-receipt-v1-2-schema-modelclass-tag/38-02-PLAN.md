<threat_model>
| Threat | Severity | Mitigation |
|--------|----------|------------|
| Fuzzy lookup assigns the wrong model class across providers | high | Use only `getCapabilityProfile("${providerId}:${modelId}")` and grep-test absence of `findCapabilityProfile` in runtime |
| Adapters are forced to report modelClass, breaking provider contracts | high | Compute in `maybeIssueReceipt`; do not change `ProviderRunResponse` or adapter options |
| Synthetic checkpoint/no-route receipts claim a real model class | medium | Return undefined when provider id or model id is empty or strict lookup misses |
| Signer failure path starts throwing because of registry lookup | medium | Keep lookup pure/synchronous and preserve maybeIssueReceipt try/catch |
</threat_model>

       ```ts

       import type { TrainingClass } from "../capabilities/profile.js";
       ```
    2. Add a local helper near `maybeIssueReceipt`:
       ```ts
       function resolveReceiptModelClass(
         route: ReceiptRoute,
         model: ReceiptModel,
       ): TrainingClass | undefined {
         if (route.providerId === "" || model.requested === "") return undefined;
         return getCapabilityProfile(`${route.providerId}:${model.requested}`)?.trainingClass;
       }
       ```
    3. Inside `maybeIssueReceipt`, compute:
       ```ts
       const modelClass = resolveReceiptModelClass(input.route, input.model);
       ```
    4. Pass the field into `createReceipt` only when defined:
       ```ts
       ...(modelClass !== undefined ? { modelClass } : {}),
       ```
    5. Do not import or call `findCapabilityProfile`.
    6. Do not add `modelClass` to `MaybeIssueReceiptInput`,
       `ProviderRunResponse`, `ProviderRunRequest`, or any adapter file.
  </action>
