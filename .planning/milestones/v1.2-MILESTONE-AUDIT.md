## Documented v1.2 Limitation

**V1.2-LIMITATION-1: Native tool_use deferred uniformly across all 7 providers.**

During Plan 19-03 implementation it surfaced that native tool_use protocols (Anthropic Messages-API `tools[]`, OpenAI Chat-Completions `tools[]`, Gemini `function_declarations`) cannot ship without modifying the `ProviderAdapter` interface in a way that risks breaking the Phase 17 INV-03 7-provider parity contract. The agent loop therefore ships **prompt-reencoded mode uniformly** across all 7 providers (openai, openai-compat, anthropic, gemini, xai, openrouter, lm-studio): conversation + tool descriptions + JSON-envelope instructions are encoded into the unchanged `ProviderRunRequest.task` field, and tool-call envelopes are parsed from the response text.

This does NOT affect any v1.2 REQ-ID. AGENT-02 explicitly requires "implements a `tool_use` protocol that works across the v1.2 provider adapters" тАФ uniform prompt-reencoded mode satisfies the contract literally. Proof: `format-tools.test.ts` `describe.each(ALL_PROVIDERS)` exercises 12 cases ├Ч 7 providers = 84 provider cases against the same handle shape.

**Carry-forward to v1.3:** When admitting native tool_use, the ProviderAdapter interface must be additively extended (e.g. optional `nativeToolUseCapability?: "anthropic" | "openai" | "gemini" | null` field) so the parity contract widens as a literal-union extension rather than breaks.
