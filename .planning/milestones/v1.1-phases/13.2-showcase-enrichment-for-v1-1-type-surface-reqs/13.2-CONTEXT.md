# Phase 13.2: Showcase Enrichment for v1.1 Type-Surface REQs - Context

**Gathered:** 2026-05-12
**Status:** Ready for planning

<domain>
## Phase Boundary

Exercise the 4 v1.1 REQ-IDs flagged as "type-surface-only" in the milestone audit so the work-inbox showcase + showcase-e2e test prove them end-to-end:

- **CONTRACT-03** (`qualityFloor` invariant): demonstrate a contract with `qualityFloor: { suite, minScore }` declared and the runtime tolerating it without enforcement (enforcement happens in `lattice eval`, not at run time)
- **COST-02** (adapter usage normalization across all three families): demonstrate normalized `Usage` shape produced by `openai`, `openai-compat` (with caller-supplied `pricing`), and `ai-sdk` adapter factories — the showcase doesn't need to make real provider calls; it can use the fake adapter's pricing-aware paths
- **EVAL-03** (judge N=3 median aggregation): wire a deterministic stub judge that returns a fixed score sequence so median aggregation is observable
- **EVAL-04** (judge cache reuse): show that a second `lattice eval` run hits the disk cache (no re-judge calls)

Out of scope: real LLM judges (deferred); any new public API; changes to receipt schema.
</domain>

<decisions>
## Implementation Decisions

### Scenario A — Quality-Floor Demonstration (CONTRACT-03)
- Add a new showcase scenario `examples/work-inbox/scenarios/quality-floor.mjs` that declares a contract with both invariants and a `qualityFloor: { suite: "examples/work-inbox/.lattice/quality-suite", minScore: 0.8 }`.
- The runtime accepts and persists the qualityFloor declaration in the receipt body but does NOT enforce it (Phase 7 routes it through the catalog hard-filter; Phase 12 enforces via `lattice eval`).
- Stdout summary explicitly notes that `qualityFloor` is declared and forwarded to the eval gate.

### Scenario B — Adapter Pricing Diversity (COST-02)
- The existing 4 showcase scenarios all use `createFakeProvider`. To exercise COST-02 without real network calls, refactor one scenario (the success scenario) so the catalog declares three model capabilities each backed by a different adapter family:
  - `fake-openai-1` modeling an `openai` adapter with bundled catalog pricing
  - `fake-openai-compat-1` modeling an `openai-compat` adapter with caller-supplied pricing (use `createOpenAICompatibleProvider({ pricing: { inputPer1kTokens: ..., outputPer1kTokens: ... } })` against a stub endpoint that returns a deterministic fake response — implementation note: provide the fake response via a test-only `fetch` mock)
  - `fake-ai-sdk-1` modeling the `ai-sdk` adapter family
- All three produce normalized `Usage` shape with `costUsd: number | null`. The showcase prints each adapter's `usage.costUsd` to stdout. The showcase-e2e test asserts all three values are non-null and follow the I-JSON serialization rule when written into the receipt body.

### Scenario C — Eval Judge Median + Cache (EVAL-03, EVAL-04)
- Wire a deterministic stub judge `stubScoringJudge(scores: number[])` into the showcase's eval invocation. Each call returns the next score from the array (cyclic). The N=3 aggregation should produce the median; the cache should suppress repeated calls.
- Showcase-e2e test new cases:
  - Case 7: run `lattice eval` with `--judge-n 3`, deterministic stub judge returning `[0.6, 0.8, 0.7]` → median is 0.7. Assert quality score in report equals 0.7.
  - Case 8: run `lattice eval` a second time with the same stub judge; cache hit short-circuits the judge calls → second run completes WITHOUT incrementing the stub judge's call counter beyond the first run's 3 calls (cache miss-then-hit pattern).

### Tests
- Showcase-e2e adds case 7 and case 8 covering the new scenarios.
- Existing 1–6 cases stay intact.
- Quality-floor scenario adds a co-located case in showcase-e2e that asserts the receipt body contains `contractHash` AND the qualityFloor invariant declaration survived canonicalization (read receipt JSON, parse `body.contractHash`, separately compute hash from sidecar `contract.qualityFloor`, assert match).

### Claude's Discretion
- The exact stub judge implementation: a closure capturing an array + index + callCount counter, exported from `examples/work-inbox/setup.mjs` or a new `examples/work-inbox/judges.mjs`.
- Whether the showcase script invokes `lattice eval` itself (no — keep the script pure-Node per Phase 13 decision; the e2e test orchestrates the eval invocations).
- Whether to refactor existing scenarios to use the three adapter families OR add a fourth scenario. Recommend adding a fourth "multi-adapter" scenario that produces a single receipt for visual simplicity, OR adding the third capability to the existing success scenario. Either is acceptable; pick the cleaner one.

### Limitations
- No real LLM is wired (judges are stubs).
- `openai-compat` adapter is exercised via a test-injected fetch mock — no real HTTP.
- The judge cache demonstration depends on deterministic input hashing; the stub judge prompt must be stable across runs.
</decisions>

<specifics>
## Specific Ideas

- The stub judge must be JSON-pure (no closures inside the receipt or sidecar). The judge identity is in the `judgePrompt` string used as a cache key; the closure with state lives only in the test/showcase process.
- For COST-02, the openai-compat fetch mock can use `globalThis.fetch = async (url) => new Response(JSON.stringify({ choices: [{ message: { content: "..." } }], usage: { prompt_tokens: 100, completion_tokens: 50 } }), { status: 200 })` in the showcase scenario file.
- Showcase-e2e test setup can monkey-patch `globalThis.fetch` for the openai-compat path.
</specifics>
