## Decisions Made

- The plan suggested aliased imports (`fsMkdir`, `fsReaddir`) for the new code. The existing top-of-file already imports `mkdir` and `readdir` from `node:fs/promises`; reusing the existing names avoids redundant alias imports and keeps the import block tidy. The `void mkdir;` no-op at the bottom of the file (kept "for future fixture fabrication") was removed since `mkdir` is now actively used by case 7's `judge-cache-case7/` setup.
- The plan's draft case-7 code declared `case7Stub: { judge: ...; get callCount(): number }`. The codebase has no `StubScoringJudge` type exported, so an inline interface `StubScoringJudge { readonly judge: Judge; readonly callCount: number }` is declared at module scope. This stays type-safe (TypeScript narrows `get callCount()` to `readonly callCount: number` at the structural level) and reuses the public `Judge` type from src/eval/judge.ts.
- Case 8's assertions use `as` casts after the `expect(...).toBeDefined()` guard: `const stub = case7Stub as StubScoringJudge; const cache = case7Cache as ReturnType<typeof createDiskJudgeCache>; const input = case7Input as JudgeInput`. This is necessary because TypeScript cannot narrow the module-scoped `let case7Stub: StubScoringJudge | undefined` based on `expect(...).toBeDefined()` (vitest's type signature doesn't perform control-flow narrowing). The `expect(...).toBeDefined()` calls remain so a regression where case 7 was skipped surfaces as a clear assertion failure, not a cryptic null-deref.
