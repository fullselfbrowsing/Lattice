# Milestone v1.3 (Public Release + Model-Aware SDK + Multi-Agent Surface) — Final Audit Report

## Verdict

**Status:** `passed` — 64 / 87 authored REQ-IDs shipped end-to-end. The remaining 23
REQ-IDs (the five canary categories `CAN` / `UNIT` / `INTEG` / `COST` / `DISPATCH`,
Phases 30–32) are **deliberately superseded**, not unmet — see below. Stable
`@full-self-browsing/lattice@1.3.0` and `@full-self-browsing/lattice-cli@1.3.0` are live
on npm with SLSA provenance attestations and a GitHub Release `v1.3.0` object.

This is the first public release milestone. It also expanded mid-flight from a
publish-and-canary scope into a model-aware SDK + multi-agent surface (Phases 33–39).

## What Shipped (13 phases, 64 REQ-IDs)

**Public release infrastructure (Phases 24–29):**
- Scope rename to `@full-self-browsing/*` (`RENAME`, `PKG`), release-hygiene docs +
  receipt-downgrade defense (`DOC`, `CRYPTO`), PR-time `ci.yml` (`CI`), tag-driven
  `release.yml` (`REL`), npm org + OIDC Trusted Publisher (`ORG`), and the rc.0 → stable
  `1.3.0` publishes with provenance + GitHub Release (`PUB`).

**Model-aware SDK (Phases 33–37):**
- Model capability registry seeded from the OpenRouter feed (~337 distinct profiles ≥200
  threshold) plus static supplements (`CAPS`); adapter quirk flags + capability
  negotiation API with live `/models` + registry fallback (`QUIRK` / `NEG`); deterministic
  prompt scaffolding helpers (`SCAFF`); opt-in output sanitizers across all 7 adapters
  (`SANITIZE`); opt-in returned tool-call validation across all 7 adapters (`VALID`).

**Receipt v1.2 + multi-agent surface (Phases 38–39):**
- Receipt schema bump to `lattice-receipt/v1.2` with optional `modelClass` tag and
  downgrade defenses (`RECEIPT12`); first-class opt-in multi-agent delegation surface —
  `defineAgent`, `runAgentCrew`, CrewDispatcher chokepoint, prompt-cache-prefix sharing,
  rate-limit-group primitive, `parentReceiptCid` receipt chaining, and the
  `examples/agent-crew/` showcase (`DELEG`).

## Superseded: Canary Validation (Phases 30–32, 23 REQ-IDs)

The original v1.3 plan included a dedicated synthetic consumer repo
`fullselfbrowsing/lattice-canary` to validate the published tarball (Layer 1 fake-provider
surface tests, Layer 2 real-provider integration under a cost ceiling, and cross-repo
publish→refresh dispatch).

**Decision (2026-06-15): descope it.** FSB, the real downstream product, is being
integrated to consume `@full-self-browsing/lattice` via the published npm package
wherever possible. A real consumer installing the published tarball validates both
packaging integrity and integration realism — strictly more credible than a synthetic
canary for the integration axis, and equivalent for packaging once FSB installs from npm
rather than a git submodule. The maintainer will feed FSB integration findings back during
v1.4 for maintenance.

**Residual risk (tracked):** FSB exercises only the slice of the public API it uses, so an
unused export could regress without detection. A thin, isolated, packaging-only canary is
parked as a possible v1.4 pickup. The cross-repo dispatch automation (`DISPATCH`) is moot
without the canary repo.
