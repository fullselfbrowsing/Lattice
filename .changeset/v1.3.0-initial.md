v1.3.0 is the first public npm release of Lattice. Both `@full-self-browsing/lattice` and `@full-self-browsing/lattice-cli` are now installable from the public npm registry under the `@full-self-browsing` scope. Earlier 1.x history shipped only as git tags on `fullselfbrowsing/Lattice` and was consumed via submodule; the per-package `CHANGELOG.md` files capture that retroactive record.

### Package Rename

- `lattice` is now `@full-self-browsing/lattice`.
- `lattice-cli` is now `@full-self-browsing/lattice-cli`.
- The CLI bin name remains `lattice`. After `pnpm add -g @full-self-browsing/lattice-cli`, the user-facing command is still `lattice --version`. Package name and bin name diverge intentionally (RENAME-2 in the supply-chain pitfalls catalog).
- The workspace dependency from CLI to runtime flips from `workspace:*` to `workspace:^` so CLI consumers track the latest compatible runtime within the minor band, not a single pinned patch (RENAME-03).

### License and Metadata

- `license: "MIT"`, `repository`, `bugs`, `homepage`, and `publishConfig.access: "public"` now ship on both publishable packages (PKG-01 through PKG-04).
- Root `package.json` keeps `"private": true` so the monorepo root is itself unpublishable, preventing `.planning/` artifacts and internal scripts from leaking into a tarball.

### CI Workflow

- `.github/workflows/ci.yml` runs install, typecheck, test, and `pnpm -r lint:packages` (publint plus arethetypeswrong) on every PR and every push to `main` (CI-01).
- Every third-party action is pinned by 40-character commit SHA, not a floating tag, to mitigate the TanStack 2026 OIDC compromise blast radius (CI-02).

### OIDC Trusted Publisher (Future)

- v1.3.0 is published via npm OIDC Trusted Publisher with provenance attestations, not a long-lived `NPM_TOKEN`. The release workflow `release.yml` lands in Phase 28. The trust tuple `(fullselfbrowsing/Lattice, release.yml, npm-publish)` is bound on npmjs.com in Phase 27 before the first publish.

### Receipt Downgrade Defense (CRYPTO-01)

- `verifyReceipt` now rejects receipts whose `body.version` is absent or equals `"lattice-receipt/v1"`. The new `VerifyResult` error kind is `schema-version-too-low`.
- This closes the Radicle 2026-03 precedent. Radicle's receipt protocol allowed the `schemaVersion` field to be optional, and an attacker could omit it to suppress downstream invariant checks against an otherwise-valid signature. The same downgrade vector applied to v1.0 receipts being submitted against a v1.1-aware verifier.
- This is a breaking semantic change. It is acceptable inside the 1.3.0 minor bump because there are no public consumers yet, every consumer of `@full-self-browsing/lattice` is starting from v1.3.0 as their first installed version. The full threat model lives in `SECURITY.md`.

See per-package `CHANGELOG.md` for the historical record.
